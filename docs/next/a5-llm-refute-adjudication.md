## 2. DEFECT 2 — "slope" over-licensing: CONFIRMED verdict-generator code defect, NEW, cross-record

### 2.1 The code defect, located

- [MEASURED] `tools/registry/verdict-gen.py` lines 555–560: `rungs` is the sorted set of distinct `config.rung` labels across eligible (+reused) runs; then

  ```python
  license_ = "none" if len(rungs) < 2 else ("sign-only" if len(rungs) == 2 else "slope")
  ```

  Nothing else feeds `license_`. It consults NEITHER the gates object (`separation_valid`, `scale_trend_rag`) NOR the frozen `extrapolation_envelope_verbatim`. `tools/registry/verdict-gen-dts.py` lines 269–272 carry the identical logic.
- [MEASURED] For a5-llm the eligible runs carry rungs {R0, R1, R2, R3} (design `scale_rungs` includes R0 = the engine/abstain-all/answer-all baseline cells — not a model-scale rung at all), so `len(rungs)=4 → "slope"`. The auditor's claim is confirmed verbatim: the license keys off distinct-rung-label count, R0 included, gate-blind and envelope-blind.
- [MEASURED] The emitted "slope" contradicts three frozen instruments simultaneously:
  1. the a5-llm envelope: "3 rungs ⇒ SIGN plus at most direction+order-of-magnitude trend language, never a slope law" (verbatim in the verdict object itself);
  2. the failed pre-registered gates: `separation_valid=false` and `scale_trend_rag=false` (per `analysis/a5_llm.py` lines 517–521 and 549–554, the trend member left the Holm family before any p-comparison — instrument-invalid);
  3. programme law's licensing PRECONDITIONS: `docs/research-plan/08-stats-and-extrapolation.md` §2.1 (line 326) licenses a scale-trend fit only when **≥3 COMPARABLE rungs** carry the same pre-registered endpoint under the same SAP — R0 is not a comparable model-scale rung, and a5-llm's registered trend machinery produced NO valid result (item 2), so there exists no licensed fit for a "slope" license to describe.
- **Correction of this document's earlier law attribution (review-directed):** [MEASURED] the governing text does NOT cap 3 rungs at sign-only, and this adjudication previously said it did — that was false. 08-stats §2.1 licenses a scale-trend fit at ≥3 comparable rungs, and G-12 (`docs/research-plan/02-data-and-reporting.md:664`) explicitly maps ≥3 `rungs_measured` → `"slope"`. §2.3's C-6 honesty note says only that a 3-rung parametric prediction interval is nearly vacuous (df=1, t₀.₉₅,₁=6.31) — inherently weak, NOT prohibited (§2.3's range cap then makes weak-PI extrapolated *statements* direction-only in practice, but that is a downstream consequence, not a license rule). The generator therefore implements the LETTER of G-12. The defect is PER-RECORD, threefold: it counts a non-model-scale rung label (R0), it is blind to §2.1's preconditions (comparable rungs + a validly produced trend result), and it is blind to the frozen envelope, which for a5-llm expressly says "never a slope law".
- [MEASURED] The contradiction is even self-exhibiting inside one artifact: `reports/auto/a5-llm/verdict-a5-llm.md` line 33 restates "never a slope law" while line 220 prints "Scale language licensed: **slope**".
- [MEASURED] Two further latent bugs in the same expression, for the fix's spec: (i) non-scale rung labels (R0 baseline cells) inflate the count; (ii) the generator's vocabulary is `sign-only` while the audit and envelopes say "sign" — harmonize when fixing.
