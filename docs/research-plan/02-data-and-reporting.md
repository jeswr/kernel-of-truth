# P2 — Data tracking & reporting system (the honesty backbone)

**Status:** design specification, 2026-07-08 (rev 4 — auditor identity upgraded per
maintainer decision: audits/red-team/paper.review run on the second-vendor Codex/GPT-5.5
model via the `codex` CLI, making G-6's run-vs-audit separation cross-vendor — see §4
identities, G-6, R-1, §7 item 2. rev 3 — verifier pass: RT-14 pseudonymous
in-record identities + unhashed `registry/identity-map.json` sidecar, schema constraint 10 —
the rule P9 §1.2 item 4 cites. rev 2 — P7 red-team pre-freeze fixes applied:
RT-5 amendment cutoff, RT-6 supersession-lineage semantics, RT-9/RT-15 audit naming +
external freeze timestamping, RT-4 decidability lint, RT-8 canonical tier caps + tier-sum
lint). Component P2 of the operational research plan
(`docs/research-plan/`). Governed by `docs/kernel-design-directives.md` (binding; esp. §3
"DATA TRACKING & REPORTING fixed UP FRONT" and the guardrails bullet). Consumes P1
(`01-hypotheses-experiments.md`) as the registry's seed content; feeds P3 (the execution DAG
reads registry status machine-readably), P4 (the skills implement the verbs defined here),
P8 (the statistical analysis plans live in P8; this system pins and *executes* them — §5.4),
and P9 (every paper/explainer claim must resolve to a registry verdict object — §5.5).
**Author:** Fable planning agent (P2), for @jeswr. Coordination: sparq-org/sparq#1683.

**Purpose in one sentence.** Make every experimental claim in this programme a *pure function
of pre-frozen declarations and append-only raw data*, so that no verdict — PASS, FAIL, NULL,
or anything else — can be produced, softened, or reframed by prose written after the numbers
are known.

**Design lineage.** This system generalises what the programme already does by hand:
`poc/x1-grounding/PREREG.md` (frozen criteria, §9 dated amendments written before unblinding,
fail-closed gates, pre-authorized fallbacks), the E-series verdict files (verbatim prereg text
quoted above the outcome; e.g. `poc/e5/results-incoming/20260707-140918-modal/verdict-e5.md`),
and the modal harness's staged-bytes sha256 manifests (`poc/modal/modal_common.py`). P2 turns
those conventions into schemas + mechanical checks so they cannot be skipped under deadline
pressure, and so an auditor (or a frontier-lab reviewer) can verify honesty without trusting
any agent's prose.

**Native-formalism note (directives §1).** Nothing in this component is RDF/OWL/SHACL-shaped.
All schemas are plain JSON Schema (draft 2020-12) over canonical JSON; identity is sha256 over
bytes, matching the encoder's own content-hash discipline. No semantic-web vocabulary appears
in any record.

## 0. Design principles (each traceable to a directive)

| # | Principle | Directive |
|---|---|---|
| P-1 | **Freeze before run.** No run whose output can feed a verdict may start unless its experiment record is FROZEN and hash-pinned. | §3 "pre-specified" |
| P-2 | **Append, never edit.** Raw results are an append-only, hash-chained log. There is no update operation; corrections are new records that reference the record they supersede. | §3 "append-only raw-results log" |
| P-3 | **Verdict = f(frozen record, log).** The verdict generator is a deterministic program; its inputs are the frozen registry record and the log; free text cannot enter the computation. | §3 "reports auto-generated from the registry" |
| P-4 | **Separation of run from audit.** The identity that produced a run's log lines may not produce its audit record. A computed PASS is not a PASS until audited. | §3 guardrails bullet |
| P-5 | **Negatives are first-class.** FAIL/NULL/INCONCLUSIVE reports are generated by the same template, committed to the same place, indexed in the same table as PASSes. | §3 "negative results committed with full statistics" |
| P-6 | **Everything pinned.** Encoder hash, corpus hashes, analysis-script hash, seeds, model revision, harness manifest — in every record. A run with any unpinned or mismatched pin is EXPLORATORY and can never feed a verdict. | §3 "hash-pinning per run" |
| P-7 | **The full metric vector or nothing.** Efficiency-relevant runs must log the complete V = {accuracy, params, memory, inference compute, training compute}; the verdict generator fails closed on a missing component. | §2 |
| P-8 | **Coverage in every verdict.** Every verdict object and report restates the M0b kernel-expressibility coverage number and its rung. | P1 common rules |
| P-9 | **Amendments are records, not edits.** Any change to a frozen declaration is a new, dated, signed amendment record with rationale. Design-scope amendments are valid only if they precede the experiment's **first final-phase run record** in the log (the raw-data-exposure cutoff — coordinator and runners lawfully see per-cell results before the formal `unblind` line); ops amendments are valid until CLOSED. | §3; x1-grounding §9 practice |
| P-10 | **Exploration is allowed, labelled, and quarantined.** Untracked tinkering is inevitable and useful; the system gives it a namespace (`phase: "exploratory"`) rather than pretending it away — but nothing in that namespace can be quoted as evidence. | honesty |

## 1. The experiment REGISTRY

### 1.1 Location and lifecycle

One JSON file per experiment at `registry/experiments/<EXP-ID>.json`, validated against
`registry/schema/experiment.schema.json`. EXP-IDs are the P1 experiment names, lowercased
(`f2`, `e9-full`, `e9-c`, `f4`, `g1`, `e8-r`, …); one registry record per *decisive
experiment*, referencing every hypothesis it decides (F2 decides HE1, HE2, HC3, HS12 — one
record, four hypothesis links).

Status lifecycle (single-direction; transitions are log events, §2.4):

```
DRAFT → FROZEN → RUNNING → READOUT → (AUDIT) → CLOSED(verdict)
                     ↘ BUDGET-HALT ↗   (resume only via amendment)
```

- **DRAFT** — editable in place. Nothing run in DRAFT can ever feed a verdict.
- **FROZEN** — the record's `frozen_sha256` (sha256 over the canonical-JSON bytes of the
  record with `status` and `frozen_sha256` fields excluded) is written into
  `registry/frozen-index.json` and committed. From this moment the file is immutable;
  `registry-check` (§4, G-2) fails CI if the bytes change while status ≥ FROZEN.
  **External freeze-timestamping (mandatory `prereg-freeze` post-step, RT-15):** the
  `frozen_sha256` (hash only, never content) is published to an external, third-party-
  timestamped surface — the public coordination issue (sparq-org/sparq#1683) at minimum,
  plus an OSF registration mirroring `frozen-index.json` when available — so
  pre-registration precedence is verifiable by an outside reviewer without trusting this
  repo's git history (closes part of R-2).
- **RUNNING / READOUT / CLOSED** — status changes are *appended to the log* and mirrored into
  the index file `registry/status.json` by tooling; the frozen record file itself never
  changes.

Canonical JSON = UTF-8, sorted keys, no insignificant whitespace, newline-terminated — the
repo's existing byte-determinism convention.

### 1.2 Experiment record schema (normative field list)

```jsonc

  "title": "string",                       // one line
  "hypotheses": ["HE1", "HE2", "HC3", "HS12"],  // P1 hypothesis IDs this experiment decides
  "prereg_doc": {                           // where the human-readable prereg lives

  "efficiency_relevant": true,               // if true, G-9: full vector V required per run
  "budget": {"usd_cap": 60, "gpu_hours_cap": 12, "wall_clock_cap_hours": 48},  // G-11
  "depends_on": ["m0b", "f0-harness"],       // P3 DAG edges (registry is the DAG's ground truth)
  "runner_constraints": {"concurrency_cap": 5, "foreground_gates": true},
  "status": "FROZEN",                        // excluded from frozen_sha256
  "frozen_sha256": "…",                      // excluded from itself
  "frozen_at": "2026-07-09T00:00:00Z",
  "frozen_by": "coordinator-1"               // pseudonymous agent identity (see §4 identities; RT-14 —
                                             //   never an account-bearing string inside hashed bytes)

```

Schema-level hard constraints (enforced by `registry-check`, not convention):

1. `endpoints` contains **exactly one** `role:"primary"` entry.
2. Every `verdict_rules[].when` leaf `{"metric": …}` resolves to a declared DV or a documented
   analysis-output field; unknown pointers fail validation at freeze time, not at readout.
3. The last `verdict_rules` entry must be `{"verdict":"INCONCLUSIVE","when":{"const":true}}` —
   the rule list is provably exhaustive.
4. `NULL` may appear as a verdict only if the primary endpoint declares
   `smallest_effect_of_interest` (the TOST bound) — non-significance alone can never be NULL.
5. If `efficiency_relevant`, the `dependent_vars` must include all five V components (name
   match against a fixed list in the schema).
6. `seeds` length ≥ 5 whenever any arm is a trained condition (`n_planned.seeds_per_trained_arm`
   present ⇒ `len(seeds) ≥ 5`).
7. `kill_criterion_verbatim` is non-empty for every experiment that can produce PASS.
8. `extrapolation_envelope_verbatim` is non-empty for every experiment whose hypotheses have a
   P1 §4b row (i.e. all of them); `analysis_plan_ref` is present and its `sha256` verifies —
   an experiment cannot freeze against an unpinned or missing stats plan.
9. **Decidability lint (Wilson-bound gates, RT-4).** Any endpoint or gate whose threshold is
   tested via a Wilson bound must be **powered for its threshold at the planned n**: the
   frozen record carries the P8 detectable-alternative number, and `prereg-freeze` fails if
   the gate is undecidable at the *expected* (not merely optimistic) rate — e.g. G2's
   0.9-precision gate requires **n = 500 gold subsumption judgments**; ~50 is undecidable at
   any realistic precision.
10. **Pseudonymous identities inside hashed bytes (RT-14).** Account-identifying material
    (account names, `<account>/…` identity prefixes, emails, profile ids, account-bearing
    repo/issue URLs) MUST NOT appear anywhere in the byte ranges covered by `frozen_sha256`,
    `prev_sha256`, or any pinned record sha — i.e. in frozen experiment records, log lines,
    amendment records, audit records, or verdict objects. Every `runner` / `author` /
    `auditor` / `frozen_by` field is a stable **pseudonym** of the form `<role>-<n>`
    (`runner-3`, `auditor-2`, `coordinator-1`); the real account↔pseudonym binding lives
    ONLY in the separate **unhashed sidecar `registry/identity-map.json`** (git-ignored in
    this repo; held privately by the maintainer — never committed publicly, never entering
    any hash). `prereg-freeze`, `log-append`, and `registry-check` scan record bytes against
    a fixed account-pattern list and fail closed (`ERR_P2_ACCOUNT_IN_RECORD`). This is the
    schema rule P9 §1.2 item 4 relies on: hashed records ship **byte-intact** into an
    anonymized review snapshot, and the anonymization overlay only ever touches
    repo-external identifiers — never a chained byte.

### 1.4 Amendment records

Path: `registry/amendments/<EXP-ID>/<seq>-<slug>.json`. Amendments never modify the frozen
file; they are separate records the verdict generator reads *in sequence order* and applies as
an overlay — and only if valid.

```jsonc
{
  "schema_version": "kot-amend/1",
  "experiment": "f2",
  "seq": 1,
  "date": "2026-07-10T12:00:00Z",
  "author": "runner-3",             // pseudonym (§1.2 constraint 10, RT-14)
  "kind": "ops",                    // "ops" | "design" | "pre-authorized-fallback"
  "rationale": "string — mandatory, free text, as long as needed",
  "patch": [ /* RFC-6902 JSON Patch against the frozen record */ ],
  "unblinding_state_at_write": "pre" // set by tooling from the log, not by the author
}
```

**Validity rules (mechanically checked, §4 G-3):**

- `kind:"design"` (touches `endpoints`, `verdict_rules`, `design.*`, `pins.analysis_script`)
  is valid **only if** the experiment's log contains no `phase:"final"` record (and a
  fortiori no `event:"unblind"` record) at the amendment's commit time. The cutoff is
  deliberately the **first final-phase run record, not the `unblind` line** (RT-5): raw
  per-cell metrics land in `results-incoming/` and the chained log continuously during the
  run, and the coordinator reviews them before committing — so design amendments must die
  at first raw-data exposure, not at first formal analysis. After the first final-phase run
  record, design amendments are invalid — the only path is a **new experiment ID**
  (`f2-rev2`) with a fresh freeze, whose registry record must link `supersedes: "f2"` and
  whose report must state why.
- `kind:"ops"` (budget, schedule, worker count, checkpoint cadence — a whitelist of JSON
  pointers baked into the schema) is valid any time before CLOSED.
- `kind:"pre-authorized-fallback"` must name the frozen-record field that pre-authorized it
  (e.g. x1-grounding's N_MS→300 rule); tooling verifies the pointer exists.

This is exactly the x1-grounding §9 discipline (four real amendments, all pre-unblinding, one
pre-authorized fallback), turned into a schema — with the design cutoff pulled forward to
first raw-data exposure.

**Supersession-lineage semantics (RT-6; binding on every gate, decision-tree expression,
Holm family, and GNG dossier):**

- **Latest-ID-with-full-lineage.** A gate or tree expression over a superseded experiment
  reads the **latest ID in the lineage** (`f2-rev2` over `f2`), but the latest verdict
  object must embed the full lineage — every predecessor ID and its verdict — and every
  rendered report and GNG dossier renders the predecessor verdicts adjacent to the current
  one. A verdict object that omits a predecessor fails `registry-check`.
- **FAIL→PASS flips are expensive.** A revision whose verdict flips FAIL→PASS is citable
  only after the full adversarial audit **plus** a committed red-team (R10) memo alongside
  the audit record; verdict-gen keeps it at PASS-PENDING-AUDIT until both exist.
- **2-revision cap.** A lineage is capped at 2 revisions; reaching the cap forces the
  **STOP-AND-PUBLISH-UNDECIDED** treatment for that mechanism (P1 §6 route) — no further
  revision of that experiment can be frozen. This closes the `-revN`-until-PASS forking
  path: each step is individually disclosed, and the chain terminates by rule.

## 2. The append-only RAW-RESULTS LOG

### 2.1 Location and physical format

One JSONL file per experiment: `results-log/<EXP-ID>.jsonl`. One line = one immutable record.
UTF-8, canonical JSON per line, newline-terminated. Lines are **hash-chained**: each record
carries the sha256 of the previous line's exact bytes, so any in-place edit or deletion breaks
the chain for every subsequent line — tampering is detectable from the file alone, without git.
Git history is the second, independent witness: `registry-check` in CI diffs the file against
the parent commit and fails unless the old content is a byte-prefix of the new (append-only in
the repository sense too).

Large artifacts (checkpoints, per-item outputs, training curves) do NOT go in the log; they
stay under `poc/<exp>/results-incoming/<runstamp>/` as today, and the log line carries their
paths + sha256s. The log is the index of record; the artifact directories are its evidence
attachments.

### 2.2 Run-record schema

```jsonc
// registry/schema/run-record.schema.json — one line in results-log/<EXP-ID>.jsonl
{
  "schema_version": "kot-log/1",
  "seq": 17,                            // strictly increasing from 0 within the file
  "prev_sha256": "…64 hex…",            // sha256 of previous line's bytes; genesis = 64 zeros
  "ts": "2026-07-11T09:14:12Z",         // UTC, tooling-stamped
  "experiment": "f2",
  "event": "run",                       // "run" | "status" | "unblind" | "audit-ref" | "note" | "supersede"
  "phase": "final",                     // "smoke" | "mock" | "exploratory" | "final"
  "runner": "runner-3",                 // pseudonymous agent identity (G-3; §1.2 constraint 10, RT-14)
  "prereg_hash": "…",                   // MUST equal frozen_sha256 of registry/experiments/f2.json
  "config": {                           // the complete resolved run configuration…
    "arm": "135M+verifier", "rung": "R1", "retry_budget": 4, "seed": 2, "…": "…"
  },
  "config_sha256": "…",                 // …and its canonical-JSON hash (dedup + audit key)
  "pins_observed": {                    // what the harness ACTUALLY loaded, measured at runtime
    "encoder_hash": "40e8c8ba…",
    "corpus_hashes": {"gloss-corpus": "36181f9b…"},
    "model_revision": "…",
    "harness_manifest": "…",
    "analysis_script_sha256": null      // null at run time; set on analysis events
  },
  "metrics": {                          // RAW numbers only — no derived verdict-adjacent stats
    "acc_covered": 0.8123,
    "acc_full_task": 0.7010,
    "n_items": 500,
    "metric_vector": {                  // G-9: mandatory when experiment.efficiency_relevant
      "accuracy": {"task": 0.7010, "kernel_covered_slice": 0.8123},
      "params": {"n_total": 135000000, "n_active": 135000000, "n_trained": 262144},
      "memory": {"peak_bytes_total_system": 2147483648},   // F0 §3.4 ledger
      "inference_compute": {"flops_per_query": 3.1e11, "latency_ms_p50": 84,
                            "latency_ms_p95": 141, "usd_per_query": 0.000021},
      "training_compute": {"flops": 0, "steps": 0, "tokens": 0}
    },
    "coverage": {"m0b_fraction": 0.62, "coverage_rung": "molecules-v0"}   // G-7
  },
  "cost": {"usd": 1.82, "gpu_hours": 0.4, "cumulative_usd": 22.10},       // G-11 feed
  "artifacts": [
    {"path": "poc/f2/results-incoming/20260711-091412-modal/results-f2-r1-arm4-s2.json",
     "sha256": "…"}
  ],
  "exit": "ok",                          // "ok" | "error" | "budget-halt" | "killed"
  "error": null
}
```

**Event types beyond `run`:**

- `status` — lifecycle transitions (`{"event":"status","from":"FROZEN","to":"RUNNING"}`).
- `unblind` — the moment analysis output is first computed over final-phase data. Written by
  the analysis tool automatically before it prints anything. Note (RT-5): the
  **design-amendment cutoff is earlier** — the first `phase:"final"` run record (§1.4,
  G-3), because raw per-cell outcomes are exposed from that moment; the `unblind` line is a
  second, later witness marking first analysis exposure, used by audits to verify ordering.
- `audit-ref` — pointer to an audit record (§4 G-6): `{"event":"audit-ref","audit_path":
  "registry/audits/f2/1.json","audit_sha256":"…"}`.
- `note` — free-text observations (anomalies, box conditions like the load-average-17
  incident in x1-grounding Amendment 4). Notes are visible to the audit but invisible to the
  verdict function.
- `supersede` — marks an earlier record as corrected: `{"event":"supersede","target_seq":12,
  "reason":"GPU OOM produced truncated metrics; rerun is seq 17"}`. The verdict generator
  excludes superseded records **only if** the superseding record's `reason` names a mechanical
  failure and the replacement uses the identical config hash; superseding a *successful* run
  is invalid (you cannot re-roll results you don't like — schema rule: `supersede` is valid
  only when the target's `exit != "ok"`).

**Write path.** All appends go through one tool, `tools/registry/log-append` (P4 skill
`log-append`): it computes `prev_sha256`, stamps `ts` and `seq`, validates against the schema,
verifies `prereg_hash` against `frozen-index.json`, refuses `phase:"final"` unless status is
RUNNING, and refuses everything if the tail hash doesn't verify (fail closed, `ERR_P2_CHAIN`).
Harnesses never write the file directly.

### 2.3 Immutability enforcement (three independent layers)

1. **Hash chain** — any edit breaks verification of all subsequent lines
   (`registry-check --chain` recomputes the full chain; runs in CI and in every `log-append`).
2. **Git append-only check** — CI compares against the parent commit: old bytes must be a
   prefix of new bytes for every `results-log/*.jsonl` (`registry-check --append-only`).
3. **Audit re-hash** — every audit record (§4) re-verifies the chain segment and artifact
   sha256s it relied on, so a post-audit rewrite invalidates the audit visibly.

### 2.4 What "raw" means (binding)

A `run` record's `metrics` block may contain only quantities computed by the run harness from
that run's own outputs — accuracies, counts, losses, latencies, byte sizes, FLOP meters. It
may NOT contain: p-values, effect sizes, TOST outcomes, Holm decisions, frontier/hull
memberships, verdict-rule inputs, or anything aggregated across seeds/arms. All of those are
*analysis outputs* and exist only in the analysis-output file produced by the pinned analysis
script at readout (§3.1 step 3). This line is what makes "the analysis plan was fixed up
front" checkable: the raw log demonstrably contains no analysis.

## 3. The AUTO-GENERATED REPORT and the verdict as a pure function

### 3.1 The verdict computation (normative algorithm)

Implemented once as `tools/registry/verdict-gen` (P4 skill `report-gen` wraps it). It is
deterministic; running it twice on the same repo state yields byte-identical output (repo
byte-determinism convention). Steps:

1. **Load & verify the frozen record.** Read `registry/experiments/<id>.json`; recompute
   `frozen_sha256`; require equality with `registry/frozen-index.json`. Mismatch ⇒ abort
   `ERR_P2_FROZEN_DRIFT` (no verdict of any kind is producible from a drifted record).
2. **Apply valid amendments** in `seq` order (validity per §1.4; an invalid amendment aborts
   with `ERR_P2_BAD_AMENDMENT`, it is never silently skipped).
3. **Select eligible run records.** From `results-log/<id>.jsonl` (chain-verified) keep
   records where ALL hold: `event=="run"`, `phase=="final"`, `exit=="ok"`, not superseded,
   `prereg_hash == frozen_sha256`, `config.seed ∈ design.seeds`, `config` levels ⊆ declared
   IV levels, and every `pins_observed` entry equals the frozen `pins` entry. Records failing
   any test are listed in the report's **Excluded-runs table** with the failed test named —
   exclusions are visible, never silent.
4. **Completeness gate.** If the eligible set does not cover `n_planned` (all declared
   arm×rung×seed cells), the verdict is **INCOMPLETE-DATA** — full stop, template still
   rendered, missing cells enumerated. (A deliberately abandoned experiment is CLOSED as
   INCOMPLETE-DATA with a `note` explaining why; it can never be quietly dropped.)
5. **Run the pinned analysis script** (`pins.analysis_script`; sha256 re-verified at
   execution) over the eligible records only. It writes
   `reports/auto/<id>/analysis-output.json` — every derived statistic (p-values, effect
   sizes, TOST bounds, Holm-corrected decisions over the declared secondary family, Pareto
   hulls, slope fits) lives here and nowhere else. The analysis script gets the eligible
   records on stdin and NOTHING else — no network, no reading other experiments' results.
6. **Write the `unblind` log line** (first time only) before emitting any analysis output.
7. **Evaluate `verdict_rules`** top-down over `analysis-output.json` using the expression
   grammar below; first rule whose `when` evaluates true fires. Any leaf `{"metric": ptr}`
   whose pointer is missing/null ⇒ verdict **INCOMPLETE-DATA** (fail closed; a bug in the
   analysis script cannot default to a verdict).
8. **Audit gate on PASS.** If the fired verdict is PASS and no CONFIRMED audit record exists
   (§4 G-6), the emitted verdict is **PASS-PENDING-AUDIT** — a distinct string that no
   downstream document may quote as PASS (the DAG treats it as not-done). Re-running
   verdict-gen after the audit record lands upgrades it.
9. **Emit** `registry/verdicts/<id>.json` (the machine verdict object) and
   `reports/auto/<id>/verdict-<id>.md` (the rendered report). Both are committed.

**Expression grammar** (the entirety of it — deliberately too weak to hide judgement in):

```
expr := {"op":"and","a":expr,"b":expr} | {"op":"or","a":expr,"b":expr}
      | {"op":"not","a":expr}
      | {"op":"gt"|"gte"|"lt"|"lte"|"eq","a":val,"b":val}
      | val
val  := {"metric": "<json-pointer into analysis-output.json>"} | {"const": number|boolean}
```

No arithmetic, no string ops, no quantifiers: anything needing computation must be computed by
the pinned analysis script into a named field, where it is inspectable and was frozen before
any data existed. Numeric comparisons are exact IEEE-754 (`gte` means `>=`); the analysis
script is responsible for rounding policy, declared in its frozen source.

**Verdict vocabulary (closed set):** `PASS`, `PASS-PENDING-AUDIT`, `FAIL`, `NULL` (TOST-bounded
equivalence), `INCONCLUSIVE`, `INSTRUMENT-INVALID`, `INCOMPLETE-DATA`, `CONSTRUCTION-ANOMALY`
(x1-grounding-style pre-analysis gate trip), `BUDGET-HALT`. P1's glossary maps onto this set
1:1. Nothing else may appear in a `verdict` field anywhere in the repo (`registry-check`
greps for impostors like "partial pass", "qualified success", "effectively passed").

## 4. HONESTY GUARDRAILS (enforceable rules)

Each rule names its **enforcement mechanism** — a check that fails closed, not a norm. The
checks live in `tools/registry/registry-check` (run in CI on every push and by the P4 skills
before/after every stage) and in the write-path tools themselves.

**Agent identities (RT-14).** Every record carries a `runner`/`author`/`auditor` string
that is a stable **pseudonym** of the form `<role>-<n>` (e.g. `runner-3`, `auditor-1`,
`auditor-2`, `coordinator-1`, `writer-1`) — never an account-bearing string (§1.2
constraint 10; `ERR_P2_ACCOUNT_IN_RECORD`). The real account↔pseudonym binding (which
account operates `auditor-2`, etc.) lives only in the unhashed sidecar
`registry/identity-map.json` (git-ignored, maintainer-held) — so no account name can ever
sit inside a `frozen_sha256`/`prev_sha256` byte range, and the P9 anonymize-by-overlay
never has to touch a chained byte. Identity is asserted at write time and cross-checked by
G-6 at the pseudonym level; **vendor-level** separation (run and audit on different
vendors' models — maintainer decision 2026-07-08: the auditor pseudonyms `auditor-<n>` and
`redteam-1` are bound to **Codex/GPT-5.5 invoked via the `codex` CLI**, an OpenAI model,
not to any Anthropic account) is declared in the identity-map and verified there by the
auditor and the maintainer, not from record bytes. This supersedes the earlier plan of a
backup Fable (Anthropic) account: a second-vendor model is a genuinely independent auditor,
materially stronger than same-vendor account separation.

- **G-1 — Prereg-before-run gate.** `log-append` refuses `phase:"final"` records unless the
  experiment is FROZEN and the record's `prereg_hash` equals the frozen-index hash; run
  harnesses receive the hash as a required CLI argument and embed it, so a harness launched
  against a DRAFT or drifted record cannot produce eligible data. *Mechanism:* write-path
  refusal (`ERR_P2_NOT_FROZEN`) + verdict-gen step-3 filter (defence in depth).
- **G-2 — Append-only, tamper-evident log.** Hash chain per §2.3 + CI byte-prefix check +
  audit re-hash. *Mechanism:* `registry-check --chain --append-only`; any failure blocks merge
  and voids downstream verdicts (verdict-gen re-verifies the chain itself).
- **G-3 — No post-hoc changes to thresholds/metrics/analysis.** Frozen records are immutable
  (hash-pinned); design changes are amendment records valid only before the **first
  `phase:"final"` run record** (raw-data exposure; the later `unblind` line is a second
  witness — RT-5); after that the sole path is a new experiment ID with `supersedes`
  linkage, a fresh freeze, and §1.4's lineage semantics (latest-ID-with-full-lineage,
  FAIL→PASS flips need full audit + red-team memo, 2-revision cap). *Mechanism:*
  `verdict-gen` step 2 validity check (`ERR_P2_BAD_AMENDMENT`) + amendment schema +
  final-run/unblind line ordering, all timestamped in the chain.
- **G-4 — Frozen analysis plan.** The analysis is a pinned script (sha256 in the frozen
  record), executed on eligible records only, in a no-network sandbox; ALL derived statistics
  live in its output file; the verdict grammar (§3.1) is too weak to compute anything.
  *Mechanism:* sha256 re-verification at execution; §2.4 raw-metrics-only schema rule (log
  lines containing p-values/effect sizes fail schema validation).
- **G-5 — Negatives committed with full statistics, same prominence.** The report template is
  verdict-agnostic: FAIL/NULL/INCONCLUSIVE render every section a PASS renders (endpoints,
  full V, coverage, excluded runs). The P3 DAG marks an experiment node complete only when
  `registry/verdicts/<id>.json` + the rendered report are committed — abandoning after a bad
  readout leaves the node visibly incomplete, and step-4 INCOMPLETE-DATA closure is itself a
  committed verdict. *Mechanism:* DAG completion predicate + verdict-gen unconditional
  rendering; `registry/status.json` global table shows every experiment and its verdict, one
  line each, autogenerated.
- **G-6 — Run-vs-audit separation.** An audit record
  (`registry/audits/<exp>/<n>.json`: auditor identity, artifacts re-hashed, re-run outputs,
  checklist per P4, outcome CONFIRMED/REFUTED/BLOCKED) is required to upgrade
  PASS-PENDING-AUDIT → PASS. Schema rule: `audit.auditor` must differ from every `runner`
  value in the experiment's eligible log records (`ERR_P2_SELF_AUDIT`). The auditor re-runs
  from pinned artifacts (adversarial checklist defined in P4). *Mechanism:* verdict-gen step
  8 + audit schema cross-check against the log. *Naming discipline (RT-9, upgraded
  2026-07-08):* the audit was originally hedged as "role-separated re-derivation on one
  operator's accounts"; it is now performed by a **genuinely independent second-vendor
  model** — Codex/GPT-5.5, invoked via the `codex` CLI — for the audit,
  adversarial-verification (red-team), and paper.review roles, which materially upgrades
  the independence claim. Documents, dossiers, pitches, and the paper describe it as
  **independent cross-vendor re-derivation (Codex/GPT-5.5)** — naming the vendor split, so
  the reader can price exactly what "independent" means here. The word "independent"
  remains reserved for this cross-vendor audit, maintainer-level audits (§7 item 2), and
  genuinely external replications — never for a same-vendor/same-operator agent audit —
  and `registry-check --citations` still flags "independent(ly) audited" wherever the
  identity-map shows the auditing pseudonym bound to the same vendor as any run identity.
- **G-7 — Coverage disclosure in every verdict.** `verdict-gen` fails closed
  (`ERR_P2_NO_COVERAGE`) if the coverage block (M0b fraction + rung) is absent from eligible
  runs' metrics or the verdict object; the report template's coverage section is mandatory.
  M0b itself is a registry experiment (`m0b`) and a P3 root gate. *Mechanism:* schema-required
  field + generator abort.
- **G-8 — Mandatory baselines incl. the kernel-as-text null.** The frozen record's
  `arms_mandatory_baselines` is linted against the P1 hypothesis table (a machine-readable
  copy at `registry/hypotheses.json`, generated once from P1 §8 and frozen with it); freezing
  an experiment for HE1 without a `kernel-as-text` arm fails (`ERR_P2_MISSING_BASELINE`).
  *Mechanism:* freeze-time lint in `prereg-freeze`.
- **G-9 — Full metric vector or no verdict.** For `efficiency_relevant` experiments, every
  eligible run must carry the complete `metric_vector` (all five components, F0 accounting);
  verdict-gen treats a missing component as INCOMPLETE-DATA. *Mechanism:* schema + step-7
  fail-closed rule.
- **G-10 — Multiple-comparison correction, fixed at freeze.** Exactly one primary endpoint
  (schema); all secondaries form one pre-declared Holm family at α=0.05, applied by the
  pinned analysis script; endpoints added post-freeze are possible only via a design
  amendment preceding the first final-phase run record (G-3/RT-5), and anything else is
  `phase:"exploratory"` (quarantined, uncitable).
  *Mechanism:* schema constraint + G-3 + G-4.
- **G-11 — Budget/kill caps.** The frozen `budget` block; `log-append` tracks
  `cost.cumulative_usd` and refuses further `final` records past the cap, writing a
  BUDGET-HALT status event; resumption requires an ops amendment raising the cap (visible,
  dated, with rationale). Tier-5 experiments additionally require a maintainer sign-off field
  (`budget.maintainer_signoff: {by, date}`) to freeze at all. **Canonical tier caps**
  (identical wherever a cap table appears — P3 GR-1, P6 §4; RT-8 reconciliation): Tier 0
  ≤ $20 · Tier 1 ≤ $80 · Tier 2 ≤ $400 · Tier 3 ≤ $400 · cumulative Tiers 0–3 ≤ $900 ·
  Tier 4 (F5) ≤ $900 · Tier 5 = maintainer-approved envelope ($2–10k). **Freeze-time
  tier-sum lint:** Σ(frozen worst-case budgets in a tier) ≤ the tier cap — `prereg-freeze`
  refuses a freeze that would break the sum (worst-case Tiers 0–3 ≈ $760 < $900).
  *Mechanism:* write-path accounting + freeze-time lint.
- **G-12 — Scale-language discipline.** `scale_language_licensed` is computed from
  `rungs_measured` count (1→"none", 2→"sign-only", ≥3→"slope"); `registry-check --citations`
  flags scale adjectives ("scales", "grows with size", "vanishes at scale") in prose about
  experiments whose license is below "slope". *Mechanism:* verdict-object field + prose
  scanner (advisory list, hard-fails on the fixed regex set).
- **G-13 — Exploration quarantine.** `phase:"exploratory"` records are welcome in the log
  (they keep tinkering honest and timestamped) but are excluded by verdict-gen and may not be
  cited in any `docs/`/`reports/` file (`--citations` treats an exploratory-only claim as a
  violation). Promoting an exploratory finding = registering a new confirmatory experiment
  and re-running under G-1. *Mechanism:* step-3 filter + citation scanner.
- **G-14 — Seed & pin integrity.** Only registered seeds and exactly-matching pins produce
  eligible records (step 3); a run with a drifted encoder/corpus/model pin is auto-demoted to
  exploratory by `log-append` (with a warning line) rather than rejected — the data is kept,
  it just cannot count. *Mechanism:* write-path demotion + step-3 filter.

**Threat table (what each guardrail kills):**

| Dishonesty pattern | Killed by |
|---|---|
| Move the goalposts after seeing data | G-3 (frozen record + first-final-run cutoff, RT-5) |
| Iterate `-revN` until a PASS lands | §1.4 lineage semantics (full-lineage rendering, audit + red-team memo on FAIL→PASS, 2-revision cap → STOP-AND-PUBLISH-UNDECIDED) |
| Re-run until it passes, report the best | G-2/G-14 (every run logged, registered seeds only, supersede restricted to `exit != ok`) |
| Bury the negative | G-5 (DAG won't close without a committed verdict report) |
| Grade your own homework | G-6 (self-audit schema error) |
| p-hack across endpoints | G-10 (one primary, frozen Holm family) |
| Quietly drop the text-null arm | G-8 (freeze-time lint) + step-4 completeness |
| Claim beyond the covered slice | G-7 (coverage in every verdict) |
| "It scales" from one rung | G-12 (license field + scanner) |
| Spin the verdict in prose | §3.3 template + `--citations` scanner |
| Tune the arm harder than baselines | disclosed via config completeness (every run's full config logged) + P4 audit checklist item; P1 "cannot be resolved by" clause binds the tree |

### 5.2 P3 (DAG) integration

- The DAG's node states are **derived, not stored**: a node `<exp>` is *runnable* iff its
  `depends_on` experiments' verdict objects satisfy the edge conditions; *running* iff the
  latest `status` event says so; *done* iff `registry/verdicts/<id>.json` exists with a
  terminal verdict AND the rendered report is committed (G-5).
- Gate conditions are expressions in the §3.1 grammar over *other* experiments' verdict
  objects — e.g. F5's gate: `f1.verdict == PASS && f3.verdict != INCOMPLETE-DATA` — declared
  in P3, evaluated by the same evaluator (one implementation, no second semantics).
- Tier-5 nodes additionally require `budget.maintainer_signoff` (G-11).
- The P1 global decision tree (§6) is itself encoded in P3 as verdict-object expressions over
  {f2, e9-full, e9-c, f4, f6, e8-r, e8-d, …} — so "TAKE-TO-FRONTIER-LAB vs
  NARROW-AND-CONTINUE vs PIVOT vs KILL vs **STOP-AND-PUBLISH-UNDECIDED**" is also computed,
  not narrated. The UNDECIDED route (RT-1) fires when no mechanism PASS exists but the
  H0-NO pattern is not met because ≥1 member is INCONCLUSIVE or INSTRUMENT-INVALID after
  its single, pre-declared replication buy.
- Gates over superseded lineages follow §1.4's **latest-ID-with-full-lineage** rule (RT-6):
  the evaluator reads the latest ID, the dossier renders predecessor verdicts adjacently,
  and the 2-revision cap routes the mechanism to STOP-AND-PUBLISH-UNDECIDED.

### 5.4 P8 (statistics) integration — the plan lives there, the report executes it

Division of labour, binding:

- **P8 owns the statistical methodology**: per-experiment analysis plans (primary test +
  justification, effect-size + CI form, α, the Holm/FDR family membership, TOST equivalence
  margins, power/sample-size justification, WLS slope-fit specs for scale claims). P2 owns
  none of that content — it owns the *mechanics that make the plan unalterable and executed
  as written*.
- **The frozen record binds to the plan twice**: `analysis_plan_ref` pins the P8 plan section
  (path + anchor + sha256 of the doc at freeze time), and `pins.analysis_script` pins the code
  that implements it. `prereg-freeze` requires both; the freeze-time review checklist (P4
  `prereg` skill) includes one human/agent judgement item — "the script implements the
  referenced plan" — recorded in the frozen record as `analysis_plan_review: {by, date}`. The
  reviewer may not be the script's author (same identity rule as G-6).
- **Execution is verbatim**: verdict-gen step 5 runs the pinned script; every statistic the
  P8 plan names must appear as a named field in `analysis-output.json`; the endpoint table in
  the report renders all of them. A P8 plan change after freeze is a design amendment (G-3
  rules apply: valid only before the first final-phase run record, else new experiment ID
  under §1.4's lineage semantics).
- **P8's cross-experiment obligations flow through P2 objects**: Holm families never span
  experiments unless the P8 plan declares a named cross-experiment family, in which case the
  family is materialised as its own registry record (`family-<name>.json`) whose analysis
  script consumes member experiments' `analysis-output.json` files by pinned sha256 — same
  freeze/unblind discipline, no informal cross-experiment corrections. The first such
  family is **`family-h0`** (P8 C-3's F-H0 Holm family, RT-13): a first-class P3 §8 node
  chain whose record must freeze **before** `a-h0` can freeze; its family is the 8
  pre-declared members, with non-read-out members scored as non-rejections (conservative),
  never a data-dependently selected subset.
- Until the P8 document lands, `analysis_plan_ref` may point at P1's "Common statistical
  rules" block + the per-hypothesis test text (P1 §§2–4) — those are already binding; P8
  refines, never relaxes, and any P8-vs-P1 conflict is resolved by amendment before freeze.

### 5.5 P9 (paper & explainer) integration — every claim resolves to a verdict object

The write-up phase (directives §7) is gated by this system, mechanically:

- **Claim manifest.** The paper repo/dir carries `paper/claims.json`: every quantitative or
  verdict-bearing claim in the paper is an entry
  `{claim_id, text, experiment, verdict_path, verdict_sha256, fields_cited: [json-pointers]}`.
  The paper source marks claims with `{{claim:<id>}}` anchors. `registry-check --paper` fails
  if: an anchor has no manifest entry; a manifest entry's `verdict_sha256` no longer matches;
  the cited verdict is `PASS-PENDING-AUDIT` or any non-terminal state; the claim text asserts
  a direction/magnitude the cited fields do not support (the same claim-regex machinery as
  `--citations`); or a scale adjective appears for an experiment whose
  `scale_language_licensed` is below "slope" (G-12 applied to the paper).
- **Envelope discipline in the paper.** Any extrapolation sentence must quote or cite the
  verdict object's `extrapolation_envelope_verbatim`; the `--paper` scanner flags
  extrapolation language (fixed regex set: "at frontier scale", "will scale", "extrapolat…")
  in sections whose claims lack an envelope citation.
- **Negatives in the paper.** `registry-check --paper` cross-checks `registry/status.json`:
  every CLOSED experiment whose hypotheses the paper's scope declaration lists must appear in
  the results section (by claim manifest entry) regardless of verdict — a FAIL/NULL cannot be
  omitted from a paper that cites its sibling's PASS. The paper's scope declaration
  (`paper/scope.json`: the list of hypothesis IDs the paper addresses) is itself frozen
  before results-section drafting begins, logged as a `status` event on a `paper` registry
  record.
- **The explainer-back** (directives §7, the plain-language maintainer report) is generated
  from the same verdict objects (a second template: what-we-found / what-it-means /
  what-scale-it-holds-at / go-no-go, each slot filled from `verdict`, `coverage`,
  `scale_language_licensed`, `extrapolation_envelope_verbatim`, and the P3-computed decision-
  tree state). It passes the same `--paper` scan. Plain language is a rendering choice;
  the numbers and verdicts it renders are the registry's.
- Both the paper and the explainer are P3 DAG nodes whose completion predicate includes
  `registry-check --paper` passing — the honesty gate on the write-up is the same executable
  check as everywhere else, not an editorial norm.

### 5.6 Migration of pre-P2 evidence (one-time, honest)

X0–X4, E1/E2/E5/E8/E9-defl, X1-grounding predate this system. They are imported as registry
records with `"retro": true` and status CLOSED, their verdicts transcribed from the existing
committed verdict files (paths + sha256s pinned in the record), and their reports NOT
regenerated — retro records exist so the DAG and status table are complete, and every
consumer can see which evidence ran under the full backbone and which predates it. No retro
record may be silently upgraded: re-running an old experiment under P2 means a new ID
(`e5-p2` style) frozen the normal way. The P1 prior-evidence ledger (§7) is the transcription
source.

### 5.7 Bootstrap order (feeds P3's first milestone)

1. Commit schemas + tools skeleton (`registry/schema/`, `tools/registry/`), with unit tests
   incl. a tamper fixture (edited mid-log line must fail the chain check).
2. Generate `registry/hypotheses.json` from P1 §8; freeze it.
3. Retro-import per §5.6.
4. Freeze `m0b` and `f0-harness` records (the P1 pre-experiment gates).
5. Freeze `f2` (the Tier-1 pivot experiment) as the first full-discipline record — its freeze

## 6. Residual risks (named, not hidden)

- **R-1 Collusion/identity spoofing.** Agent identity strings are asserted, not
  cryptographically proven; a dishonest operator controlling both runner and auditor defeats
  G-6. Mitigation (upgraded 2026-07-08): all audits run on a second-vendor model —
  Codex/GPT-5.5 via the `codex` CLI — so runner and auditor no longer even share a model
  vendor, and cross-model collusion via shared weights/quirks is off the table; the
  operator-level residual remains (one human still operates both CLIs), and the maintainer
  is the ultimate auditor for any frontier-facing claim. Accepted residual.
- **R-2 History rewrite.** A force-push can rewrite git history and re-chain the log.
  Mitigation: chain tail hashes are quoted inside audit records and verdict objects (circular
  witnesses); every freeze's `frozen_sha256` is externally timestamped (§1.1: sparq issue /
  OSF), so a rewrite cannot forge pre-registration precedence; and the remote's branch
  protection should forbid force-push to main — an explicit ask to the maintainer (open
  decision below).
- **R-3 Garbage-in metrics.** The backbone guarantees provenance and immutability, not that a
  FLOP meter is correct. That is F0's and the P4 audit checklist's job; P2 only guarantees
  the meter's output cannot be altered after the fact.
- **R-4 Analysis-script bugs frozen in.** A frozen buggy analysis produces a wrong verdict
  deterministically. Mitigation: analysis scripts ship with fixture tests (mock inputs with
  hand-computed expected outputs, the x1-grounding smoke discipline) that `prereg-freeze`
  requires to pass at freeze time; a post-unblinding bug fix is a superseding experiment ID
  with the bug documented — visible, expensive, honest.
